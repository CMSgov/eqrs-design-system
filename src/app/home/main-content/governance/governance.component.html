Governance

What is the design system?

A design system is a set of visual and coded components that can be used like Lego blocks to rapidly develop and release products that:

  • Have clean, accessible and compliant code
  • Are visually and functionally consistent
  • Serve as a single source of truth for what’s “correct” or best practice

The ESRD Design System (ESRD-DS) is a nascent system that serves the ESRD/EQRS communities. It includes:

  • A public-facing UI, found at https://cmsgov.github.io/eqrs-design-system/
  • A public Github repository, found at: https://github.com/CMSgov/eqrs-design-system

Coming soon:

  • A Sketch file containing visual assets
  • An NPM package for inclusion into your project

Who should use the design system?

ESRD-DS is primarily intended to help facilitate design and development, so the primary users are ESRD designers and engineers. However, it can also be used by testers to validate visual and code implementation.

Who should contribute to the design system?

ESRD-DS is open for pull requests from the broader Github community, per guidelines laid out in this document. We expect contributions primarily from the ESX team and EQRS developers. Suggestions, requests, and feedback should be submitted as Github issues.

Who’s responsible for maintenance?

ESRD-DS is maintained by the Shared Excellence team at ManTech.

Adding new patterns

People working on the EQRS UI should be conscious of not building “snowflake” patterns. The design system contains all the common patterns that are already in use on production, and their accepted variations.

If a designer or developer thinks they need to deviate from the design system and develop a new visual pattern, they should consider the following:

  • Does the design system have a pattern that could satisfy the need with minor modifications?
  • Is there another way to solve the problem to hand in the UI?

Developers should not develop new visual patterns without consulting a designer or the ESX team. If everyone decides that a new pattern is actually required, they should then work to develop the pattern to meet the acceptance criteria.

Acceptance criteria for new patterns:

  • An ESX-approved visual design
  • HTML and CSS to support the pattern, which have passed accessibility tests
  • If required, Angular code to support the pattern, which has passed accessibility tests
  • Appropriate documentation for when a pattern should be used and when it shouldn’t, plus how to use it (if it’s an Angular pattern)

See the Workflow below for how to submit a new pattern, once these acceptance criteria are met.

Deprecating old patterns

Any well-maintained, continuously worked-on UI will eventually no longer need certain patterns. It should periodically evaluate the patterns it contains and ensure that anything obsolete is removed.

Because ESRD-DS serves government products, and potentially multiple teams external to the core design system team, the DS team should consider whether deprecating a pattern represents a breaking change to products using the design system. As such, all efforts should be made to communicate any upcoming deprecations with a minimum of six months’ notice, before the pattern, its code and documentation are removed from the system.

Workflow

The Design System is housed on a public Github repository. While the repo is public-facing, it’s managed by a small team who are responsible for code review, merging to master, updating the UI and publishing to NPM. The master branch is protected, meaning that teammates cannot push directly to master and must submit pull requests in order to update the design system.

Submitting pull requests

Anyone with valid Github credentials can fork the repository, make changes, and submit a pull request to the DS repository for review. EQRS-DS teammates with the correct permissions can create their own branches on the repo in order to submit pull requests.

Pull requests should be code-reviewed for accessibility, compliance and other standards before approving the PR.

Merging to master

Once a PR has been appropriately reviewed and approved by at least one teammate other than the author, either that person or the author can merge the new branch to master.

Deploying to Pages

The Github repository is configured to automatically deploy the design system to Pages whenever the master branch is updated.
